# report-workflow 技能需求设计

## 1. 背景

`report-workflow` 用于生成 Luckin 工作记录体系中的日报、周报、月报和专项周总结。技能需要统一处理本地 `plan.md`、周期性进度更新、报告落盘和确认后的同步推送。

本次设计以 `skillsOfYao` 仓库为准，目标是把报告流程从“单文件汇总”调整为“按年、按类型分目录存储”，同时补齐 Git 与乐享的后续同步动作。

## 2. 设计目标

- 统一 `/daily`、`/weekly`、`/monthly`、`/shoot-weekly` 四类入口
- 统一使用 `config/system.json` 管理路径、人员名、外部目录和 MCP 配置
- 报告输出按年和类型拆分存储，便于后续检索和同步
- 周报、月报进度计算以上期报告为基线，不再只依赖当前文本聚合
- 用户确认后自动执行后续推送动作

## 3. 输出目录

本技能的本地输出结构调整为：

```text
record/
├── plan.md
├── reports/
│   └── 2026/
│       ├── daily/
│       │   ├── 2026-04.md
│       │   └── 2026-05.md
│       ├── weekly/
│       │   ├── 2026-W16.md
│       │   └── 2026-W17.md
│       ├── monthly/
│       │   ├── 2026-04.md
│       │   └── 2026-05.md
```

对应规则：

- 日报按“月文件”存储，文件名：`reports/{year}/daily/{year}-{month}.md`
- 周报按“周文件”存储，文件名：`reports/{year}/weekly/{year}-W{week}.md`
- 月报按“月文件”存储，文件名：`reports/{year}/monthly/{year}-{month}.md`
- `plan.md` 仍然是唯一工作记录输入源

## 4. 依赖配置

统一从 `config/system.json` 读取以下配置：

- `paths.recordRepoRoot`
- `paths.reportTemplatePath`
- `reportWorkflow.authorName`
- `reportWorkflow.reviewGroup`
- `reportWorkflow.lucpCreatorKeyword`
- `reportWorkflow.lexiangWeeklyRoot`
- `reportWorkflow.lexiangMonthlyRoot`
- `reportWorkflow.lexiangMcpServer`

说明：

- 乐享周报根目录、月报根目录地址、MCP 服务标识统一走配置
- 若任一必要配置缺失，先提示用户确认或补充，不允许写死默认值

## 5. 交互形式

### 5.1 统一入口

- `/daily`
- `/weekly`
- `/monthly`
- `/shoot-weekly`

### 5.2 统一交互流程

1. 识别报告类型
2. 读取配置并校验
3. 收集输入内容和时间范围
4. 生成报告草稿
5. 展示给用户确认
6. 用户确认后写入本地报告文件
7. 执行对应推送流程

### 5.3 各报交互差异

#### `/daily`

- 允许用户直接提供全量 `plan.md` 内容
- 若用户未提供，则默认以当前 `plan.md` 为准
- “直接变更”即当前工作区内容为比较基准，不要求用户额外整理

#### `/weekly`

- 周报周期固定为“上周五到本周四”
- 上周周报优先以线上周报为准；若线上不存在，再回退本地文件
- 若用户指定特殊周期，先提示这将偏离默认口径

#### `/monthly`

- 默认统计上月
- 上月月报作为本月进度与 OKR 的默认基线
- 若目标月为季度首月，或上月月报不存在 OKR，需主动向用户索取 OKR 内容或链接

#### `/shoot-weekly`

- 保留专项周总结入口
- 默认读取对应周期内容做专项归纳

## 6. 业务处置

### 6.1 公共处置原则

- 输入主源始终是 `plan.md`
- 周报、月报基于“周期内的 `plan.md` 变更”提取新增内容
- 进度更新一律以上期报告最新进度为基线
- 新增事项默认从 `0%` 开始
- 进度阶段统一采用：设计 `20%`、开发 `50%`、测试 `80%`、上线 `100%`

### 6.2 日报处置

#### 规则

- 允许用户直接粘贴全量 `plan.md`
- 若未提供，则读取本地当前 `plan.md`
- 以当前 `plan.md` 直接变更进行对比并提取今日新增完成内容
- 输出仍按工作事项聚合，补充标签、进度、说明、完成细项、文档

#### 流程

1. 获取用户提供的 `plan.md` 或读取本地 `plan.md`
2. 计算相对基线的新增内容
3. 提取今日完成项与相关文档
4. 生成日报草稿
5. 写入 `reports/{year}/daily/{year}-{month}.md`

### 6.3 周报处置

#### 规则

- 周期固定为上周五到本周四
- 周报内容不再基于日报简单聚合，而是基于该周期 `plan.md` 内容对比提取新增内容
- 原始进度以上周周报最新进度为准
- 本周期新增事项初始进度为 `0%`
- 按设计/开发/测试/上线更新到 `20% / 50% / 80% / 100%`

#### 流程

1. 确定目标周期
2. 获取线上周报，作为上期进度基线
3. 对比周期内 `plan.md` 内容，提取新增工作事项
4. 依据阶段更新进度
5. 生成周报草稿
6. 写入 `reports/{year}/weekly/{year}-W{week}.md`

### 6.4 月报处置

#### 规则

- 基于目标月份的 `plan.md` 内容对比提取新增内容
- 原始进度以上月月报最新进度为准
- 新增事项默认 `0%`
- 进度阶段同样使用 `20% / 50% / 80% / 100%`
- 月度 OKR 默认复制上月月报 OKR 内容，并根据本月日报进度更新
- 各季度第一月或 OKR 不存在时，需由用户提供 OKR 文本、链接或其它来源信息

#### 流程

1. 确定目标月份
2. 获取上月月报，提取 OKR 和工作项进度
3. 对比本月 `plan.md` 变更
4. 基于本月日报更新 OKR 进度
5. 生成月报草稿
6. 写入 `reports/{year}/monthly/{year}-{month}.md`

### 6.5 推送流程

用户确认后再执行后续推送。

#### 各场景通用

- 写入本地文件
- 执行 Git 提交
- 推送到 Git 仓库
- 更新本次 commit 信息

#### 周报同步

- 使用乐享 MCP 同步周报
- 目标目录层级：
  - `02.个人周报`
  - `{年份} - 监控组个人周报`
  - `{周期}【监控组个人周报】`
- 若对应层级不存在，需要自动创建
- 若支持复制，则优先复制上周周报后覆盖内容
- 若不支持复制，则新建文档并更新标题与内容

示例目录：

- `02.个人周报/2026 - 监控组个人周报/2026.04.10-2026.04.16【监控组个人周报】`

#### 月报同步

- 使用乐享 MCP 同步月报
- 目标目录层级：
  - `03.工作复盘`
  - `{年份}工作复盘（监控组）`
  - `姚清居 -{周期} 工作复盘 (监控组)`
- 若对应层级不存在，需要自动创建

示例目录：

- `03.工作复盘/2026工作复盘（监控组）/姚清居 -2026.03 工作复盘 (监控组)`

## 7. 关键异常处理

### 7.1 配置缺失

- 明确提示缺失键名
- 乐享根目录或 MCP 配置缺失时，先向用户确认

### 7.2 基线缺失

- 周报缺少线上周报时，回退本地上周周报
- 月报缺少上月月报时，提示用户补充进度和 OKR 基线

### 7.3 内容不足

- 当周期无新增内容时，仍可生成“无新增”草稿，但必须先让用户确认
- 月报 OKR 不存在时，不允许静默留空

## 8. 输出格式

### 8.1 日报/周报工作事项格式

```markdown
### {工作事项}
- 标签： {标签，如国内、国际化、排障、架构、AI探索、优化、答疑、其它}
- 进度： {进度，30% 或 0%->30%}
- 说明: {进度总结，如阶段性成果、上线细节等}
- 完成细项
    - {细项说明，存在多项，以日报输出为准}
- 文档
    - {文档链接，存在多项，如设计文档、上线单等}

### 其它
- {标识其它事项的概述说明}
```

约束：

- 周报的工作事项格式与日报一致，只是内容粒度为周期汇总
- `说明` 用于承接阶段性成果、反馈结论、上线细节，不再单独拆表格列
- `文档` 区块仅在存在相关链接时输出

## 9. 验收标准

- 本地输出目录符合 `reports/{year}/daily|weekly|monthly` 结构
- 日报支持“用户直接提供全量 `plan.md`”和“默认读取当前 `plan.md`”两种方式
- 周报周期固定为上周五到本周四
- 周报、月报都基于周期 `plan.md` 对比提取新增内容
- 周报、月报进度以上期报告为基线，新增项默认 `0%`
- 月报 OKR 默认继承上月，季度首月或缺失时会主动追问用户
- 用户确认后会执行 Git 推送
- 周报、月报确认后会按配置同步到乐享对应目录

## 10. 后续细化建议

- 补一份“周期内 `plan.md` 对比”的精确定义，明确时间切片和基线策略
- 补一份乐享目录创建与复制失败兜底规则
- 补一份 Git commit message 规范，区分日报、周报、月报和专项总结
