# tech-explorer 规则

## 执行流程

1. 先做结构化收集，优先给用户可直接选择的选项，并始终保留 `手动输入`。
2. 用户确定新技术名称（可选对比列表、官方文档 URL、GitHub URL）。
3. **确认 1**：展示技术名称、官方定义摘要、核心功能；优先提供直选项：`确认` / `不是` / `补充` / `手动输入`。
4. **确认 2**：展示知识库 Tag 匹配结果（按匹配度降序）；无匹配则给近似推荐。优先提供直选项：`全部对比` / `继续探索` / `对比 Top1` / `对比 Top2` / `手动输入`。
5. 深度抓取与对比：优先拉取多源可信材料，包括官方文档、官方 GitHub、预置权威来源，必要时再补论文/标准/官方工程文章。
6. 对多源内容做交叉汇总：定义优先看官方文档/规范，实现优先看官方仓库，案例与方法补充看工程文章或论文。
7. 生成报告写入 `output/reports/`，可选 `FEISHU_WEBHOOK` 推送摘要链接（由脚本支持）。

## 交互规范

### 对话侧

- 优先使用 `AskQuestion`
- 单轮问题不要过多，优先单选或少量多选
- 每一轮预设选项中都要有 `手动输入`
- 用户进入 `手动输入` 后，再追问具体文本，不要同时塞太多字段

### CLI 侧

- 默认展示编号菜单
- 菜单选项里保留 `手动输入`
- 若用户未传技术名，先展示知识库中的常用技术预设，再允许手填
- 若用户未传对比项，结合匹配结果给 `全部对比` / `继续探索` / `对比 TopN` / `手动输入`

## 参数约定

| 项 | 值 |
|----|-----|
| Tag 匹配度阈值 | 0.6（60%） |
| 确认环节建议超时 | 5 分钟（对话侧由助手计时；CLI 可用 `--yes` 跳过） |

匹配度：对「新技术 Tags」与「知识库某技术 Tags」计算 Jaccard 相似度 `|A∩B|/|A∪B|`。

## Tag 规范

共 30 个 Tag（与 `config/knowledge_base.json` 中 `tag_catalog` 一致）。新技术抽取 Tags 时：从文档摘要中按子串匹配中文 Tag 名称；未命中则依据摘要语义人工补全（对话模式由助手补全）。

## 报告格式

输出 Markdown，结构如下（标题与表格列名保持一致）：

- 全文使用中文
- 无信息章节直接省略
- 架构展示优先级：`drawio` 源文件路径 > 文字图示 > `mermaid` 图示 > 文本描述
- 对比分析默认采用四平面：控制面、数据面、执行面、治理面
- 行文风格参考论文/综述，优先使用“摘要、架构分析、对比分析、差异讨论、参考资料”
- 必须增加“可信来源汇总”或等价章节，明确来源类型、可信级别与主要贡献

## CLI 与对话一致性

- CLI：`--yes` 跳过交互确认，用于自动化。
- 对话：助手须显式完成两轮确认后再写最终报告。
- 两端都应遵循“优先直选项，保留手动输入”的交互体验。
- 两端都应遵循“中文输出、空章节省略、优先图示化架构”的产出规则。
