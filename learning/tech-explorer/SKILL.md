# tech-explorer

用于新技术探索与对比分析，产出结构化 Markdown 报告并可回写知识库。

## 何时使用

- 用户想了解某技术、框架、协议、工具或方法论
- 用户要求对比已有技术条目
- 用户希望沉淀成固定模板报告
- 用户特别关心技术架构、实现分层、物理落位或与其它技术的架构差异

## 交互原则

优先提供“直接可选项”，始终保留“手动输入”兜底，参考 superpowers 常见的结构化提问方式。

### 对话模式

优先使用 `AskQuestion` 做结构化收集，不要一上来让用户自由发挥长文本。

第一轮先收集方向，至少覆盖这些选项：

- 探索单个技术
- 对比多个技术
- 生成探索报告
- 只补充官方文档 / GitHub
- 手动输入

技术名或对比对象优先给预设选项，再保留手动输入：

- `MCP`
- `LLM`
- `Agent`
- `Cursor`
- `OpenClaw`
- `Multi-Agent`
- `Agent Harness Engineering`
- `手动输入`

如果用户选了 `手动输入`，再补一问让用户填写技术名、文档链接或对比对象。

### CLI 模式

可直接运行 `run.sh`。未传完整参数时，脚本应优先展示编号选项，并支持手动输入补充。

## 执行要点

1. 必须完成两轮确认，细节见 `RULES.md` 与 `config/confirm.json`。
2. Tag 列表与报告章节以 `RULES.md` 为准。
3. 深度抓取优先通过 Jina Reader：`https://r.jina.ai/{来源 URL}`。
4. 默认不只读取官方文档，至少要尝试汇总：官方文档、官方 GitHub、知识库中预置的权威来源；必要时再补论文/标准/官方工程文章。
5. 输出文档必须是中文。
6. 报告除基础介绍外，必须补出技术架构视角：逻辑分层优先，必要时补物理落位。
7. 若存在对比技术，必须输出架构对比表，至少覆盖：分层、主要涉及层、主要差异标记。
8. 对比分析需补标准化四平面：控制面、数据面、执行面、治理面。
9. 对于无信息、空信息、仅占位的章节，直接忽略，不输出“待补充”类空段落。
10. 架构展示优先级：`drawio` 源文件 > 文字图示 > `mermaid` 图示 > 文本描述。
11. 表述风格参考论文/综述：优先使用“摘要、架构分析、对比分析、差异讨论、参考资料”等结构与措辞。
12. 报告输出目录：`output/reports/`。

## 命令行

```bash
./run.sh
./run.sh "harness" --compare "MCP,Agent"
./run.sh "某技术" --extra-url "https://arxiv.org/abs/xxxx.xxxxx"
./run.sh "某技术" --doc-url "https://example.com/docs" --yes
```

## 文件结构

- `RULES.md`：流程、报告格式、Tag 规范
- `config/knowledge_base.json`：已有技术与 Tag
- `config/confirm.json`：确认文案与选项
- `scripts/explore.py`：抓取、匹配、交互、写报告
- `run.sh`：入口

## 维护

探索完成后可将新技术合并进 `knowledge_base.json`。
