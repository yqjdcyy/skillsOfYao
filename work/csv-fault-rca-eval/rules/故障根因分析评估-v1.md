# 故障根因分析评估 Prompt（v1）

## 输入说明

请提供以下两项输入：

- **真实根因**（运维事后复盘确认的故障根本原因，需明确为内部问题或三方问题）：`{{真实根因}}`
- **TopN 分析记录**（监控团队在故障发生时提供的前 N 条根因分析建议，按优先级排序）：`{{TopN分析记录}}`

## 评估逻辑

### 准确性判断标准
- **三方问题**：若真实根因为三方（如 CDN 异常、外部 API 故障、云厂商硬件故障、容器平台不可用等，即现有可观测体系无法直接覆盖的外部依赖），则只要 TopN 中有明确指出该三方组件或服务，即视为**准确**。
- **内部问题**：若真实根因为内部系统（如服务代码缺陷、中间件配置错误、数据库慢查询、Kafka 积压等），则只要 TopN 中有一条明确覆盖该根因（或其直接上游/下游关联组件），即视为**准确**。

### 有效性与有效率评分标准
- **有效**：TopN 中至少有一条建议能引导运维人员向真实根因方向排查（即使未直接命中，但具备相关性，例如指出异常指标、异常服务、异常链路等）。
- **有效率得分（0–100）**：
  - 若**准确**，默认得分为 **100**。
  - 若**不准确但有效**，根据 TopN 中最相关建议的引导价值打分（例如：提到了相关服务但方向错误 → 60~80；仅提及通用指标无指向性 → 30~50；完全无关 → 0）。
  - 若**无效**（完全无关、误导性强），得分为 **0**。
- **是否有效**：有效率 ≥ 50 判定为「是」，否则为「否」。

## 输出格式（请按此结构填写）

### 评估结果概览
- 是否准确：是 / 否  
- 是否有效：是 / 否  
- 有效率：[0–100]

### 评估说明概述
- **准确性原因**：[简要说明为何判断为准确或不准确，引用 TopN 中的具体内容]  
- **有效率评判依据**：[说明有效率分数的依据，如「Top3 提到数据库连接池耗尽，虽非根因但引导排查至 DB 层」]
