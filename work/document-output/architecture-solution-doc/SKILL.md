# 架构/项目方案文档输出

## 何时使用

用户要产出**技术方案、项目规划、或对外+对内双轨**类文档，并已提供（或可追问补齐）需求、背景、参考资料与初步实现思路时启用本技能。

## 母版与类型

完整条文优先打开工作区中的母版（若存在）：

`2604-故障评分跟进优化/技术模板创建.md`（路径相对 **details** 根目录），或 `.../monitor/details/2604-故障评分跟进优化/技术模板创建.md`。

**类型选择**（用户未说明时先判断并一句话确认）：

| 类型 | 采用结构 |
|------|----------|
| 需求/方案交付 | 母版「主体框架」需求/方案类 一至四 |
| 项目规划 | 母版「主体框架」项目规划类 + 定位调整八章映射 |
| 架构师深度方案 | 母版「定位调整」一至八 |
| 需对外宣讲 + 对内同步 | 母版「按需优化」内双轨模板（A/B + B.0），四预期与章节对齐句一并遵守 |

**章节标题（勿合并母版「｜」为一行标题）**  
母版中「一、需求背景｜需求梳理｜需求反馈」等用 `｜` 表示**同章内并列意图**，输出时**择一**：  
- **拆分为独立二级/三级标题**（推荐）：如「一、需求背景」下含 1.1 现状、1.2 需求梳理、1.3 范围与确认；或  
- **仅保留本章单一主标题**，并列意图全部下沉为子节。  
**禁止**把「一、需求背景｜需求梳理｜需求反馈」整串复制为文档中的章节标题。

**叙事与图示（与类型解耦的最低要求）**  
- **项目背景（原「对外主线」能力，章节推荐用此名）**：须有一节**非研发优先可读**，用多级列表写清：**背景介绍**、**用户需求**（或痛点/场景）、**核心目标**（可再缩进子项按能力域拆）、**边界与约束**（入口、优先级、非目标、关键依赖一句等）。**已给细节时也不得只堆细节不写清上述层次**。  
- **评审稿文首（默认）**：**不写**「读者与用途、本稿承载、文档定位」等元信息长列表；标题后可直接进入**项目背景**或母版第一节。与《实现细节》的**互链**用**一行**即可（如：`落地见《*-实现细节.md》`）。  
- **范围与关键结论专节**：**非必选**；小迭代、改动面小时将边界结论**并入项目背景 · 边界与约束**即可，避免重复成章。  
- **里程碑表**：**小迭代默省略**；用户明确要求或大版本再列功能包/负责人/预期。  
- **全文分层可读（评审稿/方案正文通用）**：**禁止**除项目背景外全文只剩「连续散文段 + 表」而无列表层次。版式与项目背景相同：**顶层条 + 缩进子项**；**表/图前**用 1～2 条引导。适用节示例：**干系人与协作**、**方案主轴**、**存量/本期**、**风险与依赖**、**文档索引**。实现细节篇长叙述同样优先**条文化**。  
- **重点变更标识**：评审稿中凡**本期相对存量的增量**（项目背景 · 核心目标子项、**本期概述**、**改动对照表「本期」列**、方案主轴「与存量差异」等），对**关键词句使用 Markdown 加粗**（`**…**`），便于扫读；存量列一般不加粗以免喧宾夺主。  
- **对内**：须包含 **核心环节流程**（主路径 + 关键分支；**Mermaid** `flowchart`/`sequenceDiagram` 或 `（drawio：…）`）以及 **系统调用关系**（参与方、同步/异步、数据落点；Mermaid 或简表）。  
- 平台若已提供文档页眉/元数据（作者、更新时间），**禁止**在正文重复写页眉类占位。

**迭代 / 增量需求（在评审稿或单篇方案中）**  
- 须写清：**存量操作流程**（迭代前用户与系统步骤）、**本期操作流程**（关键增量标 **【本期】**）、**改动点与影响范围**（环节 × 前端/后端/数据/角色等对照表）。  
- 评审稿中存量/本期以**步骤与环节概述**为主（可与流程图对照）；**逐步操作、检查点、接口与规则细则**放在实现细节（或研发附录）。  
- **历史需求迭代 · 示意贴图（默认推荐：流程矩阵表）**  
  - **适用**：端到端多环节、需一眼对照 **AS-IS / TO-BE**（原方式 vs 新方式，或存量 vs 本期）时。  
  - **结构**：**一张表**。首行表头：**阶段/场景**（或首列名）+ 按**真实作业顺序**排列的**环节列**（列名与流程一致，如故障统计、真实根因标识、AI 初判等）。  
  - **数据行**：**原方式**、**新方式**各一行（列名可用 **存量（迭代前）/ 本期（迭代后）** 等与全文一致）。  
  - **单元格**：每格内顺序固定为 **文字说明**（关键增量用 Markdown `**…**` 加粗）+ **原型图**（`![说明](路径)` 或「此处贴图」）；**禁止**仅贴图无说明或把多环节说明挤在同一格。  
  - **末列合并**：若末环节两版**共用**（如同一回填与统计），末列可用 **rowspan** 跨两行，格内写共用说明 + **一张**示意贴图。  
  - **表前**：**抬头说明**（1～2 条，点明本表表达「全流程 × 原/新」）。  
  - **备选 · 分阶段两行表**：多轮需求、每轮对应**不同界面域**且不宜挤入一张矩阵时，可纵向多块表；每块仍**仅两行**——第一行各列**说明/表头**，第二行各列**仅贴图**（与旧版式一致）。  
- **与 P1～Pn 两行多列的关系**：矩阵表侧重**流程/场景级**全景；**P1～Pn** 侧重**单点界面**变更。可**先矩阵后 P1～Pn**（评审信息最全），或环节少时**仅矩阵**、或仅界面点密时**仅 P1～Pn**。  
- **本期界面变更（P1…）**：默认**一块两行多列表**——**第一行**每列对应 **P1、P2…** 的**说明（表头）**（区域、**变更要点**加粗、可与实现细节 ID 互链），**第二行**每列**原型图填充**；列数与点位一致（如五列对 P1～P5）。点位少或版式受限时，可改为每点独立两行表，但**禁止**用「多行列表格把说明与贴图混在同一行」。  
- **章节命名**：小迭代可将「存量与本期」收敛为 **影响范围**（含 **原型图示意** + **改动点对照**）等，与知识库/评审习惯一致即可，不必机械沿用母版二级标题字面。  
- **实现细节**文应对照原型 ID（含矩阵表单元格、历史分块表单元格若需实现侧索引）写检查点；接口路径与 JSON 草案放在实现细节，评审稿仅指向上文。

## 方案拍板闸门（必须先做）

在输出「可当作定稿」的正文前，必须完成拍板引导；用户材料不足时**禁止**替用户虚构已定决策。

1. **列出《待拍板/待确认清单》**（条目化），至少覆盖：
   - 主线：目标、范围边界、**非目标**、关键假设
   - 技术：核心选型是否已定；若未定，列出备选与需对比维度（性能、成本、迁移、风险）
   - 交付：外部依赖、验收口径；**里程碑**大版本再问，小迭代可注明「不设里程碑表」
   - 双轨（若适用）：对外一句话目标与对内契约是否一致
2. **向用户发问**：对清单中缺失或模糊项逐条追问；可合并为一次消息中的编号问题。
3. **定稿规则**：用户未回复的拍板项，在大纲与正文中标为 `【待拍板：…】` 或 `【待确认：…】`，不得写死为最终决定。
4. 用户明确表示「按草案先出骨架、拍板后补」时：先给**带占位符的大纲**，拍板清单仍须单列。

## 执行顺序

1. 通读用户给出的需求、背景、材料、实现方案；摘录事实与引用链接/文档名。
2. 判定文档类型（上一节）；必要时问一句确认。
3. 执行**方案拍板闸门**，输出待拍板清单并追问。
4. 生成**文档大纲**（标题层级与母版章节一致，可留空小节占位）。
5. **扩展**：在已定信息范围内补全段落与表格；母版要求的技术前瞻（选型对比、迁移、风险、容量）、知识库对齐（B.0）、按需展开（降级、可观测、模块拆解）按场景写入对应节，无则标「本方案不涉及」或省略该子节并说明。
6. **自检**：**项目背景**是否含背景介绍、用户需求、核心目标、边界（或单独范围节是否多余）；文首是否**未堆砌**默认元信息；小迭代是否已**省略里程碑表**（除非用户要）；**迭代项**是否含**历史示意**（**流程矩阵表**或分阶段两行表之一）、**本期**两行多列原型表（或合规的每点两行表）、存量/本期与影响表；矩阵表是否**原/新两行**、**格内说明+贴图**、环节列与流程一致；**重点变更**是否已**加粗**；**全文**分层是否一致；对内流程图与调用关系；图示**本期**边界；若拆两篇是否**克制实现细节**；对外/对内口径一致。

## 输出习惯

- 流程与架构：优先 **Mermaid**（流程图/时序/上下文图）；复杂拓扑可另留 `（drawio：…）`。  
- **Mermaid 美观与信息层次**：可对图首使用 `%%{init: { "theme": "base", "themeVariables": { "fontFamily": "...", "lineColor": "..." }, "flowchart": { "curve": "basis", "padding": 16 } }}%%` 统一字体与线色；**flowchart** 用 **subgraph** 分层（用户/前端/后端/数据/外部）；**classDef** 区分 **存量与本期**（如今期用琥珀系填充或加粗描边）、数据存储与外部依赖；对比类需求可用**并列子图**（存量痛点 vs 本期闭环）。**sequenceDiagram** 可用 `box` 包裹浏览器/服务端/外部 participant，`actor` 表示终端用户，关键消息文案简短。  
- 不在正文写平台已覆盖的页眉/创建时间占位。  
- 与既有方案的关系写在 B.0 / 或「知识库对齐」等价节，避免重复粘贴知识库全文。

## 评审稿与实现细则拆分（按需）

当方案需过评审且实现细节较多时，**拆成两篇**。

**评审稿信息策略（默认）**  
- **首要目的**：各方**目标一致**（项目背景中已收束范围与目标即可）；**业务交接**靠干系人表 + 原型/贴图位；**风险与依赖**短表。  
- **篇幅倾向**：**概述 + 历史与本期原型/贴图占位 + 架构示意**；不写字段说明、逐步操作、长规则表。  
- **可适度展开**：仅**本期复杂核心**辅以图；其余链《实现细节》。  
- **界面**：**历史**优先**流程矩阵表**（原/新 × 环节列，格内说明+贴图；末列可 rowspan）；否则分阶段**两行表**（说明行 + 贴图行）。**本期**优先**一块两行多列**（首行各列说明、次行各列贴图）。  
- **唯二宜在评审稿写长的**：**项目背景**（背景介绍、用户需求、核心目标及子项），以及**本期复杂核心**（配图、控制篇幅）。
- **未拆成两篇时**：同一原则适用——正文保持评审向厚度，**落地细节**集中为文末「实现要点（研发）」或独立附件，避免评审稿体量大段实现描述。

| 文档 | 读者 | 内容侧重 |
|------|------|----------|
| **技术方案（评审稿）** | 评审、产品、运维、负责人 | **文首一行**链实现细节；**项目背景**；**干系人**；**方案主轴**；**存量/本期**或**影响范围**含**历史流程矩阵表**（或分阶段两行贴图表）、**本期两行多列原型表**（如 P1～P5 五列，可与矩阵二选一或并存）；**改动对照**（**本期列重点加粗**）；**架构/复杂核心** Mermaid；风险；**重点变更加粗**；**不写**字段长表、接口 JSON、逐步时序 |
| **实现细节**（可命名 `*-实现细节.md`） | 研发、Plan 工具 | 承接评审稿未展开的**全部落地细节**：与评审稿**章节互链**；**步骤级**实现要点（前/后/数/配置）；**原型 ID** 与检查点；**接口契约**、规则表、验收口径；**时序图**；**Cursor Plan 拆解**；**验收 checklist** |

评审稿文首**默认仅一行**互链实现细节（大项目如需可再加一句定位）；实现细节文首说明以评审稿为边界、不重复论证价值与架构结论。
