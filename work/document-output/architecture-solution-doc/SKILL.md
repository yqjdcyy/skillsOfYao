# 技术方案输出（architecture-solution-doc）

## 何时使用

用户需要产出**技术方案**（评审/对内），且具备或可补齐下列输入时启用。**方案思路为必填**；仅有规划链接而无思路时，先追问思路再写正文。

## 输入（三类）

| 类型 | 是否必填 | 来源 |
|------|----------|------|
| **团队规划目录** | 必填（配置） | `references.yaml` 中 `planning.title` / `planning.url` / `planning.focus`；查找顺序见下节 |
| **参照文档** | 可选 | `additional_references`；会话内用户追加的 PRD、接口说明等（写入关联文档并点对点引用） |
| **方案思路** | **必填** | 用户当次提供的实现取向、边界、关键决策草稿；缺则标 `【待补充：方案思路】` 并追问，**禁止**编造用户未陈述的主线 |

**`references.yaml` 查找顺序**（先命中先用）：  
1. 工作区根 `.cursor/architecture-solution-doc.references.yaml`  
2. 与本 `SKILL.md` **同目录**的 `references.yaml`  
3. 均无：正文标 `【待配置：规划引用】`，追问 `planning.url` 等；**禁止**编造链接  

**`references.yaml` 字段**（摘录）：`planning.title`、`planning.url`、`planning.focus`、`planning.citation_style`（可选）、`additional_references`（`title` / `url` / `role`）。正文引用规划目录须带链接；`focus` 作「与规划对齐」的检索维度，**条目级或链接对齐**，禁止空话复述全文。

## 按需项确认（必须先做）

在输出完整正文前，对**是否展开**下列块做**一次性可选项确认**：

- **技术选型**（对比维度、结论、备选淘汰）  
- **升级与迭代**（迁移步骤、兼容、回滚、发布窗口）  
- **性能与容量预估**（量级、假设、观测与压测计划）  

**确认方式（优先）**：使用 Cursor **`AskQuestion`**，每题给出互斥选项（示例）：**不展开 / 仅摘要 / 详述**；最后一项统一为 **「其他：手动说明」** 类兜底（用户可自由打字补充）。  

**兜底**：若无 `AskQuestion`，在同一条消息中用 **A/B/C/D** 编号列出选项，并写明「回复字母；或回复『手动：…』」。  

用户已明确写清「都要详述/都省略」时，可跳过重复追问，但在《待确认记录》中单行注明依据。

未定项在正文对应小节标 `【待确认：…】`，不得写死为最终决定。

## 论文式正文骨架

叙述顺序固定为**问题 → 目标 → 方法 → 证据/对照 → 边界 → 推进 → 引用**，可与工作区母版章节名合并，但逻辑不得打乱：

1. **摘要/主体介绍**：问题域、术语、本文解决与不解决的范围（各一段话级即可）。  
2. **目标与规划对齐**：业务与用户目标；与 `planning` 及 `focus` 中年度/季度/阶段表述的**对齐关系**（等价或约束）；可验收指标或观测口径。  
3. **方案概述**：主轴、关键架构决策、与存量关系；**须指向**下文架构图与对照表。  
4. **架构与前后差异**：见「架构图策略」；文字中摘要 **改前 vs 改后** 差异点列表。  
5. **原型与环节对照**：见「对照表规范」；截图可占位。  
6. **实现要点**：接口/数据/配置/流程——篇幅受「按需项」控制；详述可下沉《*-实现细节.md》并一行互链。  
7. **风险、假设与依赖**：与上下限、回滚假设挂钩。  
8. **阶段与里程碑**（可选）：由用户在按需确认中选「展开」或项目规模需要时写。  
9. **关联文档**：`planning`、`additional_references`、会话参照、实现细节路径。

**版式**：关键列表用多级条目；**改后/本期增量**关键词句 **`**加粗**`**；表/图前 1～2 条说明；平台已有页眉则正文不重复。

## 架构图策略

| 复杂度 | 要求 |
|--------|------|
| **复杂**（多子系统、多环境、强拓扑或安全域） | **必须**给出 **drawio**：工作区内 `*.drawio` 路径，正文中写 `（drawio：相对或完整路径）` 或链文件；图内或图旁说明须能区分 **改前/改后**（分色、分泳道、或「存量/本期」图例）。 |
| **简单**（单链路透传、少量节点） | **可选** **Mermaid**（`flowchart` / `sequenceDiagram`）；若已有 drawio 覆盖同一内容，避免重复两张同等信息图。 |

**前后差异**：正文中配合图示有**对照小节**（条列：存量行为 vs 本期行为，或指向 drawio 中标注）。

## 原型与环节对照表（必填其一或组合）

**表 A · 按环节一行**（默认，Markdown 表）：

| 环节/界面 | 改前（存量）描述 | 改后（本期）描述 | 变更要点 | 截图/原型 |
|-----------|------------------|------------------|----------|-----------|
| … | … | … | **本期**关键词加粗 | `![占位-环节名](path)` 或 `（截图待补）` |

**表 B · 流程全景矩阵**（环节多、需横向对齐时）：列 = 按作业顺序的环节；**两行**分别为 **改前**、**改后**；单元格内顺序：**文字说明** + **截图或占位**；末环节若两版共用可合并单元格说明。

**表 C · 界面点 P1～Pn**：列 = P1…Pn；第一行各列写**区域与变更要点**（加粗本期）；第二行各列**仅截图或占位**。

表前用 1～2 条说明表意；禁止仅贴图无文字。

## 核心事项展开规则

- **技术选型、升级迭代、性能预估**：仅当用户在「按需项确认」中选 **摘要** 或 **详述**（或手动说明）时写对应厚度；选 **不展开** 时该小节标题可保留一行「本版从略」或并入风险。  
- **详述**须含：可验证结论、假设、与方案思路一致；不得与用户已否定选项矛盾。

## 母版与文档类型（可选对齐）

若工作区存在 `monitor/details/.../技术模板创建.md`，可映射章节标题与双轨要求；母版中 **`｜` 并列标题** 勿合并为一行，须拆子节或择一主标题。

| 类型 | 结构取向 |
|------|----------|
| 需求/方案交付 | 母版需求/方案主体框架 |
| 项目规划 | 规划类 + 定位调整映射 |
| 架构深度 | 定位调整八章 |
| 对外+对内 | 双轨 A/B + B.0，口径一致 |

类型未说明时判断后**一句话确认**。

## 拍板与定稿

除按需项外，**目标、范围、非目标、关键假设、验收口径**未齐时列出《待拍板/待确认清单》并追问。用户要求「先骨架」时给大纲 + 清单，未定处保留占位符。

## 执行顺序

1. Read `references.yaml`（按查找顺序）；提取 `planning` 与可选 `additional_references`。  
2. 确认**方案思路**已给出；否则追问。  
3. **按需项确认**（`AskQuestion` 或 A/B/C + 手动兜底）。  
4. 判定复杂度 → **drawio 或 Mermaid**；复杂未给 drawio 路径时标 `【待补：drawio】` 并说明应包含的前后差异。  
5. 生成大纲（论文章节骨架），再扩写；嵌入**表 A/B/C** 与图示引用。  
6. **自检**：规划链接；方案思路是否贯穿；改前/改后是否图文一致；按需节是否与用户选项一致；截图是否占位齐全；核心事项未展开是否与选项一致。

## 评审稿与实现细节拆分

篇幅大时拆两篇：**技术方案**（对齐目标、架构对照、对照表、按需展开摘要或正文）+ **`*-实现细节.md`**（步骤、契约、JSON、checklist）。评审稿文首**一行**互链实现细节即可。

| 文档 | 侧重 |
|------|------|
| 技术方案 | 规划对齐、架构前后差异（drawio/Mermaid）、**对照表**、按需展开的选型/升级/性能、风险与关联文档 |
| 实现细节 | 与评审稿互链；落地步骤、接口与数据、验收清单 |

## Mermaid（仅简单场景可选用）

可用 `subgraph` / `classDef` 区分存量与本期；对比需求可用并列子图。复杂拓扑仍以 drawio 为准。
