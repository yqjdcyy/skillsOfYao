# 需求上线准备

与 **requirement-start-workflow** 对应：开始流程在 `details` 下建 `{YYMM}-{概述}` 与上线文档骨架。**本技能默认面向上线前**：在 `details` 内该需求目录中整理、补全对应 `*上线*.md`；**上线结束后**由用户自行将需求目录迁入 `DONE/{年}`，不在此技能范围内。

## 调用约定

用户启用本技能时，建议按顺序提供：**项目名**、**分支**、**目标上线文档**（绝对或相对路径）。

| 参数 | 是否必填 | 默认 |
|------|----------|------|
| 目标 `*上线*.md` | **必填** | 无；必须能定位到 `details/...` 下待编辑文件 |
| 项目名 | 选填 | `zeus-nebula`、`zeus-react`（两个都参与反填；若某次不涉及前端或后端，用户显式改成单仓或其它仓名） |
| 分支 | 选填 | **当前检出分支**：在各项目仓库根目录执行 `git rev-parse --abbrev-ref HEAD`，并假定**各项目名称对应仓库在该分支上同名**；若用户口头指定分支，则以指定为准并在每仓校验该分支存在 |

**执行要点**：仅用当前 Cursor 工作区时，仍须按默认或用户给出的**项目列表**，到**各仓库本地路径**分别做 `git`（路径可与用户约定，如 `.../Luckin/{项目名}`）。缺参时：**仅目标上线文件不可缺**；项目与分支按上表默认推断。

## 目录

- **主工作区（编辑对象）**：`/Users/luckincoffee/Documents/data/work/luckin/monitor/details/{YYMM}-{需求概述}/` 下的上线文档（文件名含 **上线**、**发布流程**、**上线相关** 等，与 requirement-start-workflow 命名表一致）
- **格式范例（可选）**：`/Users/luckincoffee/Documents/data/work/luckin/monitor/details/DONE/{年}/` 下历史 `*上线*.md`，仅作结构与粒度参考，非本次编辑主路径

## 工作流

1. **对齐范例（可选）**：若需统一口径，在 `DONE/{年}` 下 `glob **/*上线*.md`（必要时含 `*发布*`、`*上线相关*`），读 2～3 份与本次最接近的类型（DDL / 纯配置 / hotfix），归纳共用结构。
2. **整理目标文档**：在 **details 下当前需求目录** 中打开或创建上线文档，按「推荐章节」补全；缺项标 `略` 或留空，勿编造未核实配置。上线顺序按 **环节分层**（与 start-workflow 一致），**不用** `- [ ]`；**不必**预写大块 `<!-- ... -->`。
3. **从分支反填（多仓库须分别做）**：
   - 解析「调用约定」得到项目列表与分支名（默认两仓、分支为当前检出分支名）。同一需求还可能含 izeus、zeus-flink-job 等：**以用户本次给出的项目名为准**；未列出则仍用默认 `zeus-nebula`、`zeus-react`。**禁止**只在一个仓库跑完 git 就当全覆盖：在**每个**项目仓库中切到约定分支后，取 `<base>..HEAD` 执行 `git merge-base`、`git log --oneline`、`git diff --stat`（及必要的全量 diff），再合并写入**目标上线文档**。
   - 文档中按仓或按服务区分 **变更记录**（例如 home / analyzer 属后端仓，前端页面与路由属前端仓），避免漏记「仅有前端改动」或「仅有后端改动」的另一侧。
   - 将提交按 **模块** 与 **主题** 分组：「改动内容 / 变更事项」只写**结论级**短句（业务可见结果），**避免**长段、类名全路径、接口清单堆砌；实现细节可压缩进「版本记录 → 变更记录」或「见 diff」。
   - 从各仓 diff 汇总：**SQL（列表文件名）**、**系统配置**（**一律归入系统配置**；其中经 **Gaea 等平台下发**的 key 须在正文 **单独指明**（如小节标题或 **Gaea** 粗体行），写 **key + `+/^` + 代码块**；LSOP/环境变量/MQ 等为 **名称列表**，必要时一行备注）。**勿**把 Gaea 写成与「系统配置」并列的顶层大类。**不写** `###`、**不写**「使用场景」长段（若团队模板允许，仅用 `##系统配置` 下 **Gaea** / **其它** 两级粗体区分即可）。
   - **SQL 脚本归档**：若变更含 `.sql`，将**本次上线涉及的 sql 文件**从仓库原路径**复制**到**目标 `*上线*.md` 所在目录**（与需求文档同层）。副本文件名须符合下方「SQL 副本命名」；若仓库内原名不同，**复制时重命名**，**不改**脚本正文。上线文档「配置变更」中**脚本仅用无序列表列出规范文件名**（同目录副本）；多脚本时**序号**与上线执行顺序一致。
4. **上线顺序**：与 **requirement-start-workflow** 模板 **八环节** 同级：`配置变更/ 前` → `工单审批` → `配置变更/ 中` → `服务发布` → `功能验证` → `日志观察` → `配置变更/ 后` → `工单完成`。每环节为一级 `-`，其下 **缩进子列表** 只列**标识名**（配置 key、sql 文件名、LSOP 单元等）。**禁止**任务列表勾选（`- [ ]`）；**不写**冒号后长说明。
5. **版本记录**：与 **requirement-start-workflow** 模板一致：`- 开发分支` / `- 上线分支` / `- tag 记录` 下缩进子项；`- 变更记录` 后空行接 md 围栏代码块（语言标 `md`）或缩进子列表，按仓分段简述。

## 推荐章节（与历史文档收敛）

标题允许微调，但建议顺序如下：

| 区块 | 常见标题 | 内容要求 |
|------|----------|----------|
| 业务改动 | `# 改动内容` / `# 变更内容` / `## 变更事项` | **精简**：几条 bullet 说清「对用户/运维可见的变化」；不写 commit、少写类名与实现细节 |
| 模块进度 | `# 开发进度` | 按服务列 **模块名** 即可；非空能力可略写 |
| 配置与脚本 | `# 配置变更` / `## 配置调整` | **仅名称 + 配置**：SQL **文件名列表**（同目录副本）；**系统配置** 统归一块：**Gaea 下发项单独标明**（`- key（+或^）` + 代码块）；**其它系统项** **名称列表**（一行一项）。**禁止**再堆 `###` 与「使用场景」长文 |
| 版本 | `# 版本记录` / `## 发版内容` | 与 start-workflow：**开发分支 / 上线分支 / tag 记录** 各带缩进子项；**变更记录** 下 md 围栏或子列表，每仓数条 |
| 顺序 | `# 上线顺序` / `# 发布流程` | **环节 + 层次**：顶层八项与 start-workflow 一致；子层仅**标识名**；**不用**勾选框 |

## SQL 副本命名（需求目录内）

格式：`{序号}_{库名}_{dml|ddl}_{描述或日期}.sql`（逻辑四段：`序号`、`库名`、`dml|ddl`、`描述或日期`；**库名**可含 `_`，如 `lucky_zeus`。）

- **序号**：与实际上线执行顺序一致，`1` 最先；**若 DDL 前置**，则 `ddl` 脚本次序号小于 `dml`
- **库名**：目标库标识，如 `lucky_zeus`
- **脚本类型**：小写 **`dml`** 或 **`ddl`**
- **描述或日期**：**一段内禁止再出现下划线**（否则与分段混淆）；可用 `yyyyMMdd` 与中文连写（如 `20260407根因命中索引语义迁移`），或纯中文 / 纯日期；不用 snake_case 英文片段

示例（DDL 先执行、序号更小）：`1_lucky_zeus_ddl_20260407根因命中索引列注释.sql`、`2_lucky_zeus_dml_20260407根因命中索引语义迁移.sql`

## 配置记号（团队惯例，含 Gaea 等平台）

- Gaea 等平台下发的 key **仍属系统配置**，文档中 **单独指明** 渠道即可（勿与「系统配置」对立分章）。
- `+config.key.name`：新增配置项  
- `^config.key.name`：在已有 key 上修改（含 `home.sys.config.page.summary` 等汇总表增量）  
- 代码块语言：`json`、`sql`、`text`、`md` 按内容选择；大段 Prompt 用 ` ``` ` 包裹保持可拷贝

## 从代码提取配置与发布相关项

- **后端（Java）**：`SysConfigKeyConstants`、与需求相关的 `*Config*`、`rg` 业务 key；SQL、Dubbo/HTTP 接口变更。
- **前端**：环境文件（`.env*`、`config/*`）、接口 baseURL、路由、权限/菜单配置、CDN 或静态资源路径；与后端契约变更（字段、路径）须在「改动内容」或验证步骤中写清。
- **Flink等**：`filters/*.properties`、`topic`、并行度。
- 仅当 diff 或配置能确认时写入；不确定则「待确认」+ 仓库内路径。

## 输出约束

- 中文为主；**禁止**把未在仓库或配置平台核实的内容写成既定事实。  
- **禁止**在技能内写死密钥；文档中机器人 URL/key 沿用范例时提醒替换为正式环境。  
- **篇幅**：需求与改动描述 **能短则短**；配置区 **只保留名称与配置体**（可执行片段），不写铺陈。  
- 单份 `SKILL.md` 保持精简；编辑主文件在 `details/{需求目录}`；`DONE` 可作参考；上线顺序**按环节分层、无勾选**，版本记录仍可对齐 start-workflow。
- 含 **SQL 变更** 时：必须在**该需求目录**保留 sql **文件副本**（与 `*上线*.md` 同目录），正文与仓库源一致；**文件名**按「SQL 副本命名」规范，可与仓库内原始文件名不同。
